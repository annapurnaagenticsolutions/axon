Six design directions for AXON, compared by long-term stability and implementation scope

Option A MCP-native language

Every .ax file IS an MCP server. tool/resource/prompt keywords map directly to MCP protocol concepts.

Analog: HTML is to HTTP
Stability  ●●●○○
Build ease ●●●●○
Risk: stability tied to MCP evolution
Option B MetaMCP orchestrator

Sits above MCP servers. Routes, composes, load-balances, and fails over across the full MCP server graph.

Analog: Kubernetes is to Docker
Stability  ●●●○○
Build ease ●●●○○
Risk: framework, not language — displaceable
Option C — recommended
Dual-mode: ADL + AEL

Humans write declarative ADL (agent definitions). LLMs execute imperative AEL (act/think/observe traces). Same language, two modes.

Analog: SQL is to relational databases
Stability  ●●●●●
Build ease ●●○○○
Decoupled from any runtime — maps to stable agent theory
Option D Universal agent IR

Not a source language — an intermediate representation. Python, TypeScript, Java all compile to AXON IR. Any runtime can execute it.

Analog: LLVM IR, JVM bytecode, WASM
Stability  ●●●●○
Build ease ●○○○○
Risk: IR layers rarely get direct human adoption
Option E Open living standard

RFC-style specification. Open governance like ECMA/W3C. Multiple competing runtimes implement the AXON spec independently.

Analog: HTTP, OpenAPI, GraphQL
Stability  ●●●●●
Build ease ●●○○○
Risk: needs institutional buy-in before you can build
Option F Homoiconic / self-extending

Agents write AXON. AXON is the agent's thought format. Code equals data — agents compose and modify other agents at runtime.

Analog: Lisp homoiconicity, Smalltalk
Stability  ●●●●○
Build ease ●○○○○
Risk: safety/alignment challenges are profound

The strongest long-term design combines C + A + E: dual-mode language targeting MCP, governed as an open standard
